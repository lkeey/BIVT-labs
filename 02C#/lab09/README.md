# Лабораторная работа №9: Сетевое программирование — доска объявлений

UDP multicast-приложение «Доска объявлений» с возможностью отправки личных сообщений через unicast (+4 балла).

**Студент:** Кирюшин Д.А.
**Группа:** БИВТ-23-10
**Дата:** 17.05.2026

## Особенности реализации

1. ✅ **UDP multicast (235.5.5.1:8001, TTL=20)** — общая лента объявлений и системных уведомлений
2. ✅ **Структурированный протокол** через разделитель `|`: `AD|имя|товар|цена|chatPort` и `SYS|текст`
3. ✅ **Фоновый поток приёма** через `Task.Run` — главный поток UI не блокируется
4. ✅ **`Dispatcher.UIThread.Post`** вместо `Form.Invoke` (PDF §7) — корректное обновление UI из фонового потока
5. ✅ **`ReuseAddress`** — на одной машине можно запустить несколько экземпляров
6. ✅ **Корректное завершение** через флаг `alive` и обработку `ObjectDisposedException`
7. ✅ **+4 балла:** личные сообщения продавцу через **unicast** UDP (`MSG|отправитель|текст`)
   - Второй сокет `chatClient` на порту 0 (ОС выбирает свободный)
   - Словарь `sellerContacts` хранит `(IP, port)` каждого продавца
   - Кнопка «Написать продавцу» активна только при выбранном чужом объявлении
   - Модальный диалог `WriteToSellerWindow` для ввода текста
   - Второй фоновый поток `ChatReceiveLoop` принимает личные сообщения и показывает их в MessageBox

## Протокол сообщений

| Тип | Формат | Транспорт |
|---|---|---|
| Объявление | `AD\|имя\|заголовок\|цена\|chatPort` | multicast 235.5.5.1:8001 |
| Системное | `SYS\|текст` | multicast 235.5.5.1:8001 |
| Личное | `MSG\|имя_отправителя\|текст` | **unicast** → IP+chatPort продавца |

Поле `chatPort` в AD-сообщении нужно, чтобы получатель знал, куда отправлять личное сообщение. IP-адрес продавца берётся из `remoteIp` метода `client.Receive(ref remoteIp)`.

## Ответы на вопросы PDF

### 1. Чем отличается UDP от TCP? Почему выбран UDP?

UDP — без установки соединения, без гарантии доставки и порядка пакетов, быстрее. TCP — с подтверждением каждого пакета и переотправкой потерянных. Для широковещательной рассылки объявлений идеально подходит UDP: один пакет за один вызов `Send` уходит сразу всем участникам группы, при этом потери единичного объявления не критичны.

### 2. Что такое порт?

Порт (0–65535) — числовой идентификатор приложения внутри хоста. IP определяет машину, а порт — конкретный процесс, которому ОС адресует входящий пакет. Один компьютер может одновременно слушать сотни портов.

### 3. Multicast vs broadcast?

Broadcast (`255.255.255.255`) идёт всем в подсети, включая программы, которым он не нужен — это шум. Multicast (`224.0.0.0–239.255.255.255`) — виртуальная группа: пакет получают только те, кто явно вызвал `JoinMulticastGroup`. В работе используется `235.5.5.1`.

### 4. Что будет без `JoinMulticastGroup`?

Программа сможет **отправлять** в группу, но **принимать** её пакеты не будет — ОС их отфильтрует.

### 5. Почему `Receive` нельзя в главном потоке?

`Receive` — блокирующий вызов: он останавливает поток до прихода пакета. В главном потоке UI это «заморозит» окно — оно перестанет перерисовываться и реагировать на клики. Поэтому приём выносится в `Task.Run`.

### 6. Зачем `MethodInvoker` / `Dispatcher.UIThread.Post`?

Элементы UI можно изменять только из главного UI-потока. Если попробовать обновить `adsListBox` напрямую из фонового потока — будет исключение. `Invoke`/`Post` ставит делегат в очередь UI-потока, и обновление выполняется уже там.

### 7. Зачем `alive = false` до `Close()`?

`Close` синхронно прерывает `Receive` исключением `ObjectDisposedException`. Фоновый поток ловит его и проверяет флаг: если `alive == false`, значит сокет закрыт намеренно — поток молча завершается. Если бы флаг ставился после `Close`, был бы интервал, когда `Receive` уже прерван, а флаг ещё `true` — исключение бы пробросилось наружу как ошибка.

### 8. Что такое `ObjectDisposedException`?

Исключение, возникающее при обращении к закрытому/освобождённому объекту. В нашем случае — при работе с уже закрытым `UdpClient` после вызова `Close`. Это штатный механизм пробуждения заблокированного на `Receive` потока.

### 9. Почему разделитель `|`, а не пробел?

Названия товаров и системные тексты могут содержать пробелы («Велосипед детский», «Иван вошёл на доску»). Символ `|` крайне редко встречается в реальных строках, что делает разбор `Split('|')` однозначным.

### 10. Что в `remoteIp` после `Receive`?

`IPEndPoint` отправителя — его IP-адрес и номер исходящего порта. Именно из `remoteIp.Address` берётся IP продавца для unicast-отправки личного сообщения.

### Доп. вопрос: почему личные сообщения через unicast, а не multicast?

Multicast — broadcast по подписке: пакет получают все участники группы. Если отправлять «Сколько торг?» через multicast, его прочитают все находящиеся в сети покупатели — это нарушает приватность личной переписки и создаёт сетевой шум. Unicast (`UdpClient.Send` на конкретный IP+порт) гарантирует, что пакет дойдёт **только** до одного получателя — продавца. IP мы знаем из `remoteIp` его AD-сообщения, порт — из поля `chatPort` в том же сообщении.
